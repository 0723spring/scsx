# 开发流程与验收标准

## 1. 总体开发原则

本项目时间较短，开发原则是：

```text
先跑通闭环，再替换真实 OCR，再优化效果。
```

不要一开始就追求完整算法或复杂页面。第一目标是让系统可以稳定演示。

## 2. 阶段划分

### 阶段一：项目骨架

目标：搭好前后端基本结构。

需要完成：

- 创建项目目录。
- 创建后端 FastAPI 入口。
- 创建前端静态页面。
- 配置 `requirements.txt`。
- 创建 `uploads/` 和 `outputs/`。
- 完成 `/api/health`。
- 前端能显示基础页面。

验收标准：

- 后端可以启动。
- 访问 `/api/health` 返回正常。
- 前端页面可以在浏览器打开。

### 阶段二：上传与 mock 识别

目标：不接真实 OCR，先让上传和返回结果跑通。

需要完成：

- 前端选择图片。
- 前端显示原图预览。
- 前端发送 `multipart/form-data` 请求。
- 后端保存上传文件。
- 后端 mock OCR 返回固定文本。
- 字段提取模块从 mock 文本中提取字段。
- 后端返回统一 JSON。
- 前端展示字段和 OCR 文本。

验收标准：

- 上传一张图片后，页面能展示识别结果。
- 即使图片内容不同，也能先返回 mock 结果。
- 接口返回格式符合 `docs/02_api_spec.md`。

### 阶段三：脱敏图片生成

目标：后端生成并返回脱敏图片。

需要完成：

- 使用 Pillow 打开上传图片。
- 根据 mock OCR 的 box 遮挡姓名、手机号、地址区域。
- 保存脱敏图片到 `outputs/`。
- FastAPI 挂载 `outputs/` 静态目录。
- 前端通过 `masked_image_url` 展示脱敏图片。

验收标准：

- 接口返回 `masked_image_url`。
- 浏览器可以访问脱敏图片。
- 前端可以显示脱敏图片。
- 图片上至少有敏感区域被遮挡。

### 阶段四：虚拟数据生成

目标：生成多张可测试的快递面单。

需要完成：

- 准备至少 1 张快递面单模板。
- 编写生成脚本。
- 随机生成姓名、手机号、地址、快递单号。
- 生成图片到 `dataset/generated/`。
- 生成标签到 `dataset/labels/`。

验收标准：

- 至少生成 20 张虚拟面单。
- 每张图片有对应 JSON 标签。
- 字段内容不使用真实个人信息。
- 生成的图片可以被系统上传识别。

### 阶段五：PaddleOCR 接入

目标：将 mock OCR 替换成真实 OCR。

需要完成：

- 安装并测试 PaddleOCR。
- 在 `ocr_service.py` 中封装 `run_paddle_ocr()`。
- 将 PaddleOCR 返回结果转换为统一 OCR 格式。
- 保留 mock OCR 作为兜底模式。
- 使用虚拟面单测试真实识别效果。

验收标准：

- 可以通过配置切换 mock OCR 和 PaddleOCR。
- PaddleOCR 模式下接口仍返回相同 JSON 结构。
- 至少对 3 张虚拟面单完成真实 OCR 识别。

### 阶段六：整理与答辩材料

目标：形成最终可交付内容。

需要完成：

- 补充 README。
- 整理接口说明。
- 截取运行效果图。
- 整理系统流程图。
- 准备演示数据。
- 准备 PPT。
- 准备答辩稿。

验收标准：

- 可以从启动后端、打开前端、上传图片到展示结果完整演示。
- PPT 中包含系统背景、架构、流程、关键实现、效果展示和总结。

### 阶段七：旋转鲁棒性与图像预处理增强

目标：在主流程稳定后，增加轻微倾斜图片的处理能力。

需要完成：

- 生成 rotated 数据集。
- rotated 数据单独放在 `dataset/generated/rotated/`。
- rotated 标签放在 `dataset/labels/labels_rotated.json`。
- rotated 数据不混入主 train/val/test。
- 新增 `backend/preprocess.py`。
- 实现轻微倾斜校正。
- 在 `/api/recognize` 中增加 `enable_preprocess` 参数。
- 前端增加“图像校正”开关。
- 使用 rotated 图片测试校正前后的 OCR 和字段提取效果。

验收标准：

- 至少有 60 张 rotated 测试图片。
- rotated 标签中 `metadata.boxes_valid=false`。
- 后端可以在启用预处理时输出校正后的图片或脱敏图。
- 前端可以传入是否启用预处理。
- 答辩时可以说明 rotated 数据用于鲁棒性测试，不影响主流程演示。

## 3. 推荐开发顺序

严格建议按以下顺序推进：

```text
1. 后端 /api/health
2. 后端 /api/recognize 接收文件
3. 后端 mock OCR
4. 后端字段提取
5. 前端上传与预览
6. 前端展示识别结果
7. 后端生成脱敏图片
8. 前端展示脱敏图片
9. 虚拟面单生成脚本
10. PaddleOCR 接入
11. 优化字段提取和打码效果
12. 旋转鲁棒性测试和图像校正
13. 文档、截图、PPT、答辩稿
```

## 4. 代码协作规则

### 4.1 模块边界

不要把所有逻辑都写进 `main.py`。

推荐边界：

| 文件 | 负责内容 |
| --- | --- |
| main.py | FastAPI 应用、路由、静态目录、CORS |
| schemas.py | 响应结构 |
| services.py | 主业务流程串联 |
| ocr_service.py | OCR 和 mock OCR |
| extractor.py | 字段提取 |
| masker.py | 图片脱敏 |
| file_utils.py | 文件保存和校验 |
| app.js | 前端请求和页面状态 |
| style.css | 页面样式 |

### 4.2 接口优先

前后端联调以前，先固定接口格式。接口字段不要随意改名。

核心字段名固定为：

```text
receiver_name
phone
raw_phone
address
tracking_number
ocr_texts
masked_image_url
processing_time_ms
```

### 4.3 可配置 OCR 模式

建议用一个变量控制 OCR 模式：

```text
OCR_MODE=mock
OCR_MODE=paddle
```

即使 PaddleOCR 环境有问题，系统也能退回 mock 模式完成演示。

### 4.4 错误处理

以下情况必须有错误提示：

- 未选择图片。
- 文件格式不支持。
- 后端服务未启动。
- 上传失败。
- OCR 处理失败。
- 后端返回 `success=false`。

## 5. 测试清单

### 5.1 后端测试

- 访问 `/api/health`。
- 上传 jpg 图片。
- 上传 png 图片。
- 上传非图片文件。
- 上传损坏图片。
- 上传大图片。
- 检查 `uploads/` 是否保存原图。
- 检查 `outputs/` 是否生成脱敏图。
- 检查 JSON 字段是否完整。

### 5.2 前端测试

- 未选择图片时，识别按钮不可用或提示选择图片。
- 选择图片后能显示预览。
- 点击识别后有 loading 状态。
- 成功后展示结构化字段。
- 成功后展示 OCR 文本。
- 成功后展示脱敏图片。
- 失败后显示错误信息。

### 5.3 数据测试

- 至少 20 张生成面单。
- 至少 5 张用于最终演示。
- 字段内容不重复太严重。
- 手机号格式正确。
- 快递单号格式正确。

## 6. 演示流程

答辩现场建议按以下顺序演示：

```text
1. 打开前端页面
2. 展示一张虚拟快递面单
3. 上传图片
4. 展示原图预览
5. 点击开始识别
6. 展示识别出的结构化字段
7. 展示 OCR 完整文本
8. 展示脱敏图片
9. 说明后端接口和处理流程
10. 说明虚拟数据生成原因和隐私保护意义
```

## 7. PPT 建议结构

PPT 可以按以下结构组织：

```text
1. 项目背景：快递面单存在大量个人隐私信息
2. 项目目标：识别关键信息并进行隐私保护
3. 总体架构：前端、后端、OCR、字段提取、图片脱敏
4. 数据来源：虚拟快递面单生成
5. 核心流程：上传、OCR、抽取、脱敏、展示
6. 接口设计：/api/health 与 /api/recognize
7. 关键实现：统一 OCR 格式、正则提取、图片打码
8. 系统展示：原图、识别结果、脱敏图
9. 测试结果：多张面单测试
10. 总结与展望：可接入更强 OCR、批量识别、历史记录
```

## 8. 最终交付物

最终仓库至少包含：

- 后端代码。
- 前端代码。
- 虚拟数据生成脚本。
- 若干生成样例图片。
- 项目 README。
- 接口说明文档。
- 设计说明文档。
- PPT。
- 演示截图。

## 9. 最低可交付版本

如果时间非常紧，最低版本只需要做到：

- 前端可上传图片。
- 后端返回 mock 识别结果。
- 前端展示字段。
- 后端生成脱敏图。
- 前端展示脱敏图。

这已经能形成完整演示闭环。PaddleOCR 可以作为“已预留接口并可扩展接入”的增强项。
